原因很简单：这两段代码属于两个不同的二进制入口文件，它们各自独立配置自己的启动逻辑和日志行为。

**区别说明**
- `src/main.rs` 是默认二进制程序 `rust-crud-api` 的入口，里面显式设置了环境变量 `RUST_LOG` 的默认值，然后再调用 `env_logger::init()`。
- `src/bin/hyperliquid_demo.rs` 是另一个二进制程序的入口，只调用了 `env_logger::init()`，并未设置 `RUST_LOG`；它主要用 `println!` 输出信息，没有使用 `log` 宏，所以对日志级别不敏感。

**为什么 main.rs 有而 demo 没有**
- 每个二进制（`src/main.rs` 和 `src/bin/*.rs`）都是独立的可执行程序，彼此不会共享 `main.rs` 里的初始化代码。
- `main.rs` 需要更细的日志控制（服务端），所以提供了默认 `RUST_LOG`；`hyperliquid_demo.rs` 是演示程序，逻辑更简单，依赖 `println!`，通常不需要默认日志配置。

**如果你希望 demo 也有同样的默认日志**
- 在 `src/bin/hyperliquid_demo.rs` 里添加下面这段在 `env_logger::init()` 之前执行：
  ```rust
  use std::env;

  // 设置日志级别（如果未设置）
  if env::var("RUST_LOG").is_err() {
      env::set_var("RUST_LOG", "info,actix_web=info,rust_crud_api=debug");
  }

  env_logger::init();
  ```
  注：`env::set_var` 是安全函数，不需要 `unsafe` 包裹；`main.rs` 里的 `unsafe { ... }` 可以去掉。

**也可以用命令行直接设置**
- 不改代码就能控制日志级别：
  - `RUST_LOG=info cargo run --bin hyperliquid_demo`
  - `RUST_LOG=rust_crud_api=debug,actix_web=info cargo run --bin rust-crud-api`

**建议做法**
- 为避免重复，可在 `src/lib.rs` 或 `src/utils/` 抽一个小工具函数，比如 `init_logging_with_default()`，两个二进制都调用它，统一默认行为。这样更易维护。
